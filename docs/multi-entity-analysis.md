## 一、核心理解

**机器人的ERP账号 = 商务人员视角的账号，不需要切换。** 不存在"决定用哪个金蝶账号"的问题。

真正的情况是：**一个CRM订单可能拆成多个ERP销售订单，每个ERP订单的 `FSaleOrgId`（销售组织）不同**，分别归属不同的公司主体，但都通过同一个金蝶账号（商务B的账号）来制单和操作。

```
商务B用一个金蝶账号登录
  ├→ 创建深圳主体的ERP单（FSaleOrgId=100）
  └→ 创建香港主体的ERP单（FSaleOrgId=106）
      都通过B的账号操作，只是销售组织字段不同
```

## 二、为什么需要拆单？

### 场景：深圳接单，部分从香港海外仓出

```
CRM订单：深圳公司接单，100台设备
  ├→ 50台 → 深圳仓有货 → 深圳FSaleOrgId=100 → ERP单A
  └→ 50台 → 深圳没货，香港欧洲仓有货
       → 香港FSaleOrgId=106 → ERP单B
       → 深圳→香港：需要内部调拨（深圳"转"给香港）
```

不是"切换账号"的问题，而是**深圳主体和香港主体之间需要有一笔内部交易**，让香港主体"接收"这批货的货权，才能从香港主体的仓库发出。

### 涉及三层关系

```
                外部客户
                    ↑ 下单（付款给深圳）
        ┌───────────┴───────────┐
        │                       │
    深圳主体                 香港主体
  (FSaleOrgId=100)        (FSaleOrgId=106)
        │                       │
        │ ① 内部调拨/结算        │
        │ 深圳→香港：转货权      │
        │ 香港→深圳：付调拨价    │
        │                       │
    深圳仓                 欧洲仓
        │                       │
        └───────────┬───────────┘
                    ↓
              终端客户收货
```

## 三、系统需要处理的三个环节

### 环节1：调货（深圳→香港的货权转移）

金蝶层面如何实现"深圳把货转给香港"？

我对金蝶内部调拨流程不太确定，我的理解有两种可能：

**方案A：做内部调拨单**
```
深圳主体 → 创建内部调拨单（深圳→香港）
         → 深圳仓减少库存
         → 香港仓增加库存（同时在香港主体下记账）
```

**方案B：直接走销售**
```
深圳主体 → 创建销售订单（卖给香港公司）
         → 价格=内部调拨价
         → 香港主体的海外仓发出给终端客户
```

**这取决于金蝶多组织的实际配置**——一期需要B或金蝶管理员确认。

但不管哪种方式，系统需要的核心数据是一样的：
- 调拨的物料+数量
- 调拨价格（内部价）
- 调拨方向（深圳→香港）
- 关联的CRM订单号

### 环节2：结算（深圳和香港之间怎么算钱）

跨主体调货意味着财务上深圳和香港之间有一笔账：

| 项目 | 说明 |
|------|------|
| 深圳收入 | 客户付的钱（外部销售） |
| 深圳成本 | 调拨给香港的货的成本 |
| 香港收入 | 深圳付的调拨款 |
| 香港成本 | 从海外仓发货的物流成本 |
| 对账周期 | 按月/按批次？ |

**结算规则需要财务确认**——比如按成本价调拨？还是按成本+加成？还是按市场价折扣？

### 环节3：通知干系人

| 对象 | 通知时机 | 内容 |
|------|---------|------|
| 深圳商务(B) | 识别到需要调货时 | "订单XXX缺货，海外仓有库存，是否从香港主体调货？" |
| 香港商务 | 调货确认后 | "深圳主体向您发起调货申请：物料XXX*50台" |
| 深圳财务 | 调货完成后 | "本月跨主体调拨汇总：深圳→香港，物料XXX，金额XXX" |
| 香港财务 | 调货完成后 | "本月跨主体调拨：香港←深圳，物料XXX，金额XXX" |
| 物流 | 调货确认后 | "订单XXX从欧洲仓发货，请安排" |

## 四、一期处理策略

一期睿数试点，以**单一主体（深圳）为主**。跨主体调货场景：

| 场景 | 一期处理 |
|------|---------|
| 深圳接单→深圳仓发 | ✅ 自动走完全流程 |
| 深圳接单→需要从香港调货 | ⚠️ **生成异常→B人工处理**，不自动拆单 |
| 跨主体结算 | ❌ 走现有线下流程 |
| 多主体通知 | ❌ 不自动通知 |

### 一期需要预留的

```python
# CrmSalesOrder / MiddlePlatformOrder 扩展
sales_entity: str          # 下单主体（从CRM解析，如"SZ"/"HK"）
fulfillment_entity: str    # 出货主体（初始同下单主体，调货时变为"HK"）

# 新增 InterEntityTransfer（仅建表，一期不使用）
id: str
source_entity: str         # 调出主体
target_entity: str         # 调入主体
crm_order_id: str          # 关联CRM订单
material_json: str         # 物料明细
transfer_price: Decimal    # 调拨价格
status: str                # Draft
```

## 五、二期自动化设计

```
二期跨主体自动流程：

CRM订单 → 中台
  → 判断：仓库归属哪个主体？
    ├─ 全部归深圳 → 正常流程
    └─ 部分归香港 → 触发跨主体流程
        ├─ 系统自动拆单：
        │   ├─ 子单A：深圳主体(FSaleOrgId=100) → 深圳仓发
        │   └─ 子单B：香港主体(FSaleOrgId=106) → 欧洲仓发
        ├─ 自动创建调拨记录：深圳→香港
        ├─ 自动通知：深圳商务+香港商务+双方财务+物流
        └─ B确认后 → 两单分别走ERP制单
```

金蝶侧的自动操作需要确认：
1. 内部调拨单API是否支持创建（`Stk_TransferBill` 的 Save/Submit）
2. 多组织下 `FSaleOrgId` 切换是否有额外限制
3. 跨组织的价格/币种转换规则
