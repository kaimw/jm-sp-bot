# 跨主体调货、结算与通知 — 流程设计

> 背景：下单主体（接单的）和出货主体（发货的）可能不是同一家公司
> 尤其是海外销售场景，深圳接单但货从香港的海外仓出

---

## 一、业务场景还原

### 场景A：深圳接单，从深圳仓出（最简单的）

```
深圳公司接单 → 深圳仓有货 → 从深圳仓发 → 一个主体完事
```

### 场景B：深圳接单，从香港主体调货出（跨主体）

```
深圳公司接单 → 深圳仓没货 → 沟通后发现香港的海外仓有货
→ 流程：
  1. 深圳主体：从原单删除这部分物料
  2. 香港主体：新建一个销售订单，从海外仓发货
  3. 深圳→香港：内部调拨（深圳"卖给"香港）
  4. 香港→终端客户：实际发货
```

### 场景C：深圳接单，深圳和香港各出一部分（混合）

```
深圳公司接单100台 → 深圳仓有50台、香港欧洲仓有50台
→ 流程：
  1. 深圳主体：原单保留50台（从深圳仓发）
  2. 香港主体：新建一个单，50台从欧洲仓发
  3. 深圳→香港：内部调拨（50台的货权转移）
```

这就是原文中B演示的案例：
> "这个单有12台因为欧洲仓有货，调整成欧洲仓发货。这时候我们又要在香港账号做一个单"

---

## 二、本质：一笔外部销售 + 一笔内部交易

每一笔涉及跨主体的订单，实际拆分为：

```
┌─────────────────────────────┐
│        终端客户订单          │
│   (外部：客户→深圳公司)      │
└──────────┬──────────────────┘
           │ 需要跨主体调货时
           ▼
┌─────────────────────┐     ┌─────────────────────┐
│  深圳主体            │     │  香港主体            │
│  (下单方/接单方)     │     │  (出货方/发货方)     │
├─────────────────────┤     ├─────────────────────┤
│ 对外：← 客户下单     │     │ 对外：→ 海外仓发货   │
│ 对内：→ 向香港调货   │     │ 对内：← 深圳调货需求 │
│ 留存：部分自己发     │     │ 发出：从海外仓出      │
└──────────┬──────────┘     └──────────┬──────────┘
           │                            │
           └──────────┬────────────────┘
                      ▼
           ┌─────────────────────┐
           │  内部调拨/结算       │
           │  深圳→香港：付调货价  │
           │  (或走内部对账)       │
           └─────────────────────┘
```

## 三、流程设计

### 3.1 中台如何识别需要跨主体

| 判断条件 | 说明 | 数据来源 |
|---------|------|---------|
| CRM订单的发货仓库 | 如果仓库=海外仓 | CRM附件/订单字段 |
| 客户/渠道类型 | 独立站/亚马逊等海外渠道 | CRM客户信息 |
| 库存归属 | 深圳仓有货还是海外仓有货 | 库存表(Excel导入) |
| 结算方式 | 是否涉及海外结算 | CRM settlement_method |

**判断逻辑：**
```
CRM订单进来
  → 查库存（库存表）
  ├─ 深圳仓够 → 单主体，正常流程
  └─ 深圳仓不够/或订单指定从海外仓发
      ├─ 需要香港主体出货 → 进入跨主体流程
      └─ 通知B确认调货方案
```

但这里有一个关键问题——**谁来决定是调货还是换方案？** 原文中B说：
> "海外仓的库存如果不够的话，就会沟通销售，你让他去借货"

所以这个决策本身就需要人介入。系统能做的是一次判断和生成通知。

### 3.2 跨主体处理的三步

```
Step 1: 中台识别出"需要跨主体调货"
  → 生成异常任务："该订单需要从香港主体出货"
  → 通知：深圳商务(B) + 香港商务

Step 2: B确认调货方案后，触发系统自动操作
  → 深圳主体：
      ├─ 保留深圳仓可发的部分
      └─ 删除需要香港发部分，生成"拆单确认"
  → 香港主体：
      ├─ 自动创建香港主体的中台订单
      └─ 走香港的ERP制单(Save→Submit→Audit)

Step 3: 结算与通知
  → 记录深圳→香港的调拨关系
  → 通知：深圳财务 + 香港财务（调拨金额、对账依据）
  → 两个主体各自走后续的发货流程
```

### 3.3 系统需要记录的信息

```
调拨记录 (InterEntityTransfer):
  ├─ source_entity: 深圳（发起方）
  ├─ target_entity: 香港（接收方）
  ├─ source_order_no: 深圳的订单号
  ├─ target_order_no: 香港的订单号
  ├─ material: 物料+数量+调拨价格
  ├─ status: 待确认/已确认/已完成
  ├─ created_by: B用户
  └─ created_at: 时间

两单关联:
  深圳订单 ←→ 香港订单（通过 transfer_id 关联）
  一笔外部销售拆成了两笔ERP单，但客户看到的是一个订单
```

## 四、一期的处理策略

### 一期不做自动跨主体

一期睿数（深圳主体）试点，场景相对简单：

| 场景 | 一期处理 | 负责人 |
|------|---------|--------|
| 深圳仓发货 | 自动走完全流程 | 系统 |
| 需要香港调货 | **生成异常任务，B手动处理** | B人工 |
| 跨主体结算 | **不处理**，走现有线下流程 | 财务 |

### 一期需要预留的数据结构

```python
# MiddlePlatformOrder 新增
entity_code: str         # 主体编码（SZ/HK/WH）
inter_entity_ref: str    # 跨主体关联单号（如果是调货来的/去的）
transfer_id: str         # 调拨记录ID

# 新增表 InterEntityTransfer
id: str
source_entity: str       # 源主体
target_entity: str       # 目标主体
source_order_id: str     # 源主体订单ID
target_order_id: str     # 目标主体订单ID
material_json: str       # 调货物料明细
transfer_price: Decimal  # 调拨价格
status: str              # Draft/Confirmed/Completed
notified_roles: str      # 已通知角色
created_at: datetime
```

### 二期实现自动化

```
二期：
  ├─ 主体映射表：维护各主体的ERP组织ID、币种、通知人
  ├─ 仓库归属表：每个仓库归属哪个主体（决定调货方向）
  ├─ 调拨价格规则：内部调拨价的来源（成本价+加成/市场价折扣）
  ├─ 自动拆单：识别需要跨主体后，自动拆为两个子订单
  ├─ 自动通知：深圳财务+香港财务+双方商务
  └─ 对账报表：每月/每周跨主体调拨汇总
```

## 五、通知干系人

跨主体调货涉及多角色通知：

| 通知对象 | 通知时机 | 通知内容 |
|---------|---------|---------|
| 深圳商务(B) | 识别到需要跨主体时 | "订单XXX需要从香港主体出货，请确认" |
| 香港商务 | 确认调货后 | "深圳主体向您发起调货：物料XXX*12台，请安排发货" |
| 深圳财务 | 调货确认后 | "本月跨主体调拨：深圳→香港，金额XXX" |
| 香港财务 | 调货确认后 | "本月跨主体调拨：香港→深圳，金额XXX" |

---

## 六、关键决策点：谁来确认调货方案

从对话中可以看出，**调货方案不是系统能自动决定的**——需要B和销售沟通：
- 是借货（深圳后面还回去）？
- 还是直接调拨（深圳卖给香港）？
- 运费谁承担？
- 价格怎么算？

因此一期的合理做法是：
1. **系统识别**出"深圳仓不够、海外仓有"
2. **生成异常任务**给B："订单XXX缺货，海外仓有库存，请确认是否从海外仓发货"
3. **B确认后**，系统生成调货记录，通知干系人
4. **发货流程**由各主体各自完成
