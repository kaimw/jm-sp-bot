# 商务 AI Agent 中台 — 边界条件与自动化决策设计

> 不从"人工流程环节"出发，而从"决策点/边界条件/判断依据"出发
> 中台的设计目标：替代人工的重复判断，不是模拟人工的每一步操作

---

## 一、全局决策树：中台自动化判断主线

```
CRM 审批完成订单进入中台
  │
  ├─ 步骤A：前置条件检查（替代原来秦鸿的人工审核）
  │   ├─ CRM合同审批状态 = "已完成"？  → 否 → 阻断，通知销售
  │   ├─ 销售邮箱是否存在？            → 否 → 通知销售补充
  │   └─ 订单附件是否齐全？           → 否 → 标记缺失，继续
  │
  ├─ 步骤B：字段完整性与主数据匹配（替代原来B的人工核对）
  │   ├─ 必填字段完整？               → 否 → 阻断
  │   ├─ 物料名→金蝶料号（别名匹配）  → 失败 → 阻断，通知B确认
  │   ├─ 客户主数据映射               → 失败 → 阻断
  │   ├─ 金额一致性检查               → 异常 → 阻断
  │   └─ 附件一致性检查               → 异常 → 阻断
  │
  ├─ 步骤C：库存与发货条件判断（替代原来B的人工多轮沟通）
  │   ├─ 海外单 → 查海外仓库存
  │   │   ├─ 够 → 步骤D
  │   │   └─ 不够 → 标记缺货异常，通知销售
  │   └─ 国内单 → 国内库存（一期无数据 → 跳过，仅提示）
  │
  ├─ 步骤D：ERP 制单（自动执行，替代原来B的手工录入）
  │   ├─ Save(制单) → Submit(提交) → Audit(审核)
  │   └─ 失败 → 创建 ExceptionCase
  │
  └─ 步骤E：发货通知 + OMS 下推
      ├─ 生成发货通知 → 拆单预览 → 确认 → OMS
      └─ 一单多收货人 → 附件解析 → 分拆→多个发货通知

变更/修改走分支路径（详见下文）
```

## 二、各决策点的边界条件详细分析

### 决策点A：前置条件检查（替代秦鸿的审核）

**人工现状：** B做单前→发给秦鸿确认→秦鸿查合同审批、回款、物料一致性→确认OK后B才继续

**中台设计：**

| 检查项 | 数据源 | 判断逻辑 | 阻断级别 | 通知对象 |
|--------|--------|---------|---------|---------|
| 合同审批状态 | CRM订单 `approval_status` | = "Approved" 或 "已审批" | CRITICAL | 销售 |
| 合同/回款一致性 | CRM金额字段 | 回款金额 >= 合同金额？(一期仅状态判断) | HIGH | 商务(秦鸿) |
| 销售邮箱 | CRM `sales_user_email` | 不为空 | HIGH | 销售 |
| 一期范围 | `v2_crm_phase1_scope_json` | 部门/渠道匹配 | CRITICAL | — |

**边界条件：**
- 如果合同审批未通过但销售申请了特批 → 需要秦鸿手动确认放行（系统提供"特批标记"）
- 如果是一期范围外的订单 → 不走中台，维持现状（邮件/人工）

---

### 决策点B：物料别名匹配（替代B的手工比��）

**人工现状：** B看到CRM的"三维扫描仪"→查自己的料号表→问销售是基础款/升级款→确认料号

**中台设计：**

| 匹配层级 | 方法 | 置信度 | 处理 |
|---------|------|--------|------|
| ① 精准匹配 | 附件提取关键词 → 精确命中料号 | 100% | 自动通过 |
| ② 别名匹配 | CRM名称 → 别名库 → 唯�的料号 | 90-99% | 自动标记来源 |
| ③ 语义匹配 | LLM语义比对 → 候选料号列表 | 60-89% | 自动取最高分，生成异常请B确认 |
| ④ 无法匹配 | 完全无法识别 | 0% | 阻断，创建 ExceptionCase |

**边界条件：**
```
CRM产品名 → 别名库匹配
  ├─ 唯一命中 → 自动 ✅
  ├─ 多个候选 → 取评分最高的（可配阈值）
  │   ├─ 最高分 > 90 → 自动确认 ✅
  │   └─ 最高分 < 90 → 生成"请确认"异常 ⚠️
  └─ 无命中 → LLM语义匹配
      ├─ 有候选项 → 同"多个候选"分支
      └─ 无候选项 → 阻断 ❌
```

**边界条件细节（摘自原文）：**
- 同名不同版本：基础款/升级款/豪华款，需区分 → 别名库需维护版本维度
- 同物料不同插头：美规/欧规/英规/澳规/4规 → 别名需包含规格维度
- 老物料清库存：个别老物料剩单规版本 → 别名需标记状态（"仅剩美规"等）
- 海外英文名：B每天做"英语"翻译 → LLM需要支持中英文语义匹配
- 定制物料（换包装）→ 研发出独立料号 → 系统无法自动处理，转异常

### 决策点B2：订单类型自动识别（备货 vs 销售订单）

**背景：** 备货订单和销售订单都走CRM，但有不同的预审规则。中台需要自动识别订单类型，走不同的规则链。

**自动识别依据：**

| 判断维度 | 销售订单 | 备货订单 |
|---------|---------|---------|
| CRM订单渠道 | 渠道销售/大客户 | 独立站/亚马逊/FBA补货 |
| 订单金额字段 | 有（合同金额 > 0） | 无或为0 |
| 附件数量 | 至少1个（PI/合同） | 无附件 |
| 特殊关键字 | — | "备货"/"补货"/"FBA" |
| 审批流程 | 合同审批 | 无合同审批或简化审批 |

**判断逻辑：**
```
CRM订单进入中台 →
  ├─ 有附件(PI/合同) + 金额>0 → 销售订单
  ├─ 金额=0 + 无附件 → 备货订单
  └─ 模糊情况 → 标记"需人工确认类型"
```

**两种订单的预审规则差异：**

| 规则 | 销售订单 | 备货订单 |
|------|---------|---------|
| ① 必填字段 | 全字段 | 客户+物料+数量（简化） |
| ② 一期范围 | 检查 | 检查 |
| ③ 客户映射 | 检查 | 检查 |
| ④ 金额正数 | 检查 | **跳过** |
| ⑤ 金额一致性 | 检查 | **跳过** |
| ⑥ 明细存在性 | 检查 | 检查 |
| ⑦ 已知SKU | 检查+别名匹配 | 检查+别名匹配 |
| ⑧ BOM匹配 | 检查 | **跳过** |
| ⑨ 合同金额一致性 | 检查 | **跳过** |
| ⑩ 附件一致性 | 检查 | **跳过** |
| ⑪ 库存可用性 | 检查 | 检查 |
| 价格填充 | 从CRM传入 | **从产品价格表自动计算** |

**边界条件：**
- 订单类型识别在前置条件检查**之前**执行，决定走哪条规则链
- 备货订单的ERP制单，`FPrice`字段从中台维护的产品价格表取
- 如果CRM既有金额又有附件，但实际是备货 → 销售可标记后重新入队

### 决策点C：库存与发货条件（替代B的多轮沟通）

**人工现状：** B在制单前→问海外仓→不够→找销售借货→借不到→判断能否国内发→确认运费→调PI→...

**中台设计：** 不依赖ERP库存接口，提供**人工维护+系统判断**的半自动模式

**库存数据来源：Excel导入**
- 管理台提供库存导入页面，B每周从海外仓负责人拿到库存表 → 上传Excel → 系统自动解析入库
- 国内仓同理，B每周从仓库（张燕）拿到国内库存表 → 上传即可
- 导入的数据写入 `ProductInventorySnapshot` 表
- 系统记录每次导入的时间和操作人，支持审计追溯

**库存预审逻辑：**
```
订单区域判断 (order_region.py)
  │
  ├─ 海外单 → 查 ProductInventorySnapshot（海外仓）
  │   ├─ 库存 >= 需求量？ → ✅ 步骤D
  │   └─ 库存 < 需求量？
  │       ├─ 库存为 0 → 标记"海外仓缺货"，通知销售
  │       └─ 部分满足 → 标记"海外仓仅X台可用"，通知B
  │
  └─ 国内单 → 查 ProductInventorySnapshot（国内仓）
      ├─ 库存 >= 需求量？ → ✅ 步骤D
      └─ 库存不足 → 标记缺货
          └─ 通知生产/仓库
```

**库存边界条件：**
- 海外仓：B每周Excel导入 → 库存数据新鲜度=周级 ✅
- 国内仓：B每周Excel导入（同海外仓流程）✅
- 借货流转：B需要找销售、找亚马逊借 → 系统无法自动，生成异常通知销售
- 生产备货：库存不足时推给生产 → 通过异常机制+通知
- 分仓决策：B在管理台选择哪个仓发出，系统记录

**管理台需要新增：**
1. 库存导入页面：上传Excel → 预览 → 确认导入（支持模板下载）
2. 库存查询页面：按仓库/物料查看当前库存快照及最近更新时间

### 决策点D：ERP制单（新自动化流程）

**边界条件：**

| 条件 | 自动处理 | 人工介入 |
|------|---------|---------|
| 预审全部通过 | 自动Save→Submit→Audit | — |
| 多主体（深圳/香港账号） | 根据销售组织字段选 `FSaleOrgId` | 一期单一主体，数据模型预留 |
| 制单Save失败（字段缺漏） | 解析错误信息，生成ExceptionCase | B人工修正基础资料 |
| 提�失败 | 解析错误，生成ExceptionCase | B人工处理 |
| 审核失败（审批流阻塞） | 生成ExceptionCase | B查金蝶审批流 |

**金蝶Save所需的必填字段映射（已验证）：**
```
CRM字段                   → 金蝶字段
客户编号(CustomerID)      → FCustomerID.FNumber
销售组织(OrgID)           → FSaleOrgId.FNumber
部门(DeptID)              → FSaleDeptId.FNumber
物料(料号)               → FSaleOrderEntry[].FMaterialId.FNumber
数量                      → FSaleOrderEntry[].FQty
销售员                   → FSalerId.FNumber
销售类型                 → F_UXYO_Assistant.FNUMBER
单据日期                  → FDate
备注(特殊需求摘要)        → FNote
```

**价格来源说明：**
ERP制单需要价格字段。CRM传过来的价格是**销售价**，金蝶需要的是**成本价或内部价**。两者口径不同，不可混用：
- 正常销售订单：CRM销售价存入中台订单，但ERP制单时不传销售价（金蝶销售订单的价格字段由财务定义）
- 备货订单：不传价格，由中台根据维护的**产品内部价表**自动填充
- 因此管理台需要**产品物料与价格维护页面**（详见决策点F）

### 决策点D2：产品物料与价格维护（新增管理台功能）

**背景：** 备货订单需要中台自动计算价格，正常销售订单的别名匹配也需要维护。需要一个统一的产品主数据维护入口。

**功能清单：**

| 功能 | 用途 | 面向用户 |
|------|------|---------|
| 产品SPU列表 | 查看所有成品，维护别名库 | 商务(B) |
| SKU管理 | 查看具体规格（美规/欧规/4规等） | 商务(B) |
| 价格维护 | 维护每个SKU的**内部成本价**（备货订单用） | 商务(B) |
| 别名维护 | 为每个SPU添加/删除业务别号（CRM名称映射） | 商务(B) |
| Excel导入 | 批量导入价格和别名 | 商务(B) |

**边界条件：**
- 价格按SKU+渠道维度维护（海外/国内价格不同？）
- 别名按SPU维度维护（一个SPU多个别号）
- 价格变更记录审计日志
- 一期先支持单渠道默认价，二期扩展多渠道价

### 决策点F：备货订单（之前定为"不纳入"，修正为一期纳入）

**背景：** 备货订单是独立站/亚马逊等渠道的补货需求。原文B说"不走销售"，但爸爸说可以走CRM，只是不需要传价格和附件，由中台自动计算。

**备货订单 vs 正常订单的差异：**

| 对比项 | 正常销售订单 | 备货订单 |
|-------|------------|---------|
| CRM流程 | 标准下单+审批 | 走CRM，但不传价格+附件 |
| 合同/回款审核 | 需要 | **跳过** |
| 价格来源 | CRM销售价（用于审核） | **产品价格表自动计算** |
| 附件（PI/合同） | 必须 | 不需要 |
| 特殊需求处理 | 同 | 同 |
| 发货流程(OMS) | 同 | 同 |

**备货订单在预审规则中的差异：**
- 跳过规则：合同金额一致性(⑨)、金额正数(④)？部分
- 保留规则：物料匹配(⑦)、必填字段(①)、客户映射(③)、库存(⑪)、特殊需求
- 新增规则：价格从产品价格表填充

**管理台产品价格维护页面的具体字段：**
```
SPU名称：产品线（如"三维扫描仪"）
SKU编码：具体规格（如"CR-scan 01 基础款 4规"）
内部成本价：XXXX元
渠道价格（可选）：各渠道销售参考价
别名列表：CRM中可能的叫法
最近更新人：XXX
最近更新时间：XXXX-XX-XX
```

### 决策点E：改单/变更处理（替代B的人工判断）

**核心边界条件：订单的当前状态**

| 当前状态 | 改单类型 | 中台处理 | 人工介入 |
|---------|---------|---------|---------|
| 已归档/已闭环 | 任何修改 | **新建中台订单**+新建ERP单 | — |
| DELIVERY_READY/OMS_PENDING | 修改物料/数量 | 撤销发货通知→重审→ERP修改(Save覆盖) | B确认撤销 |
| DELIVERY_READY/OMS_PENDING | 修改地址 | 撤销发货通知→更新→重发 | — |
| DELIVERY_READY/OMS_PENDING | 新增物料 | 原单叠加→ERP修改(Save增量) | — |
| 已部分发货(SHIPPED) | 加新单 | 仅新物料走完整流程 | — |
| 已部分发货(SHIPPED) | 改已发部分 | **阻断**：已发不可改，通知销售 | 需商务特殊处理 |
| OMS_BLOCKED/异常 | 任何修改 | 需先解决OMS阻塞 | B先处理异常 |

**CRM修改检测机制：**
```
crm_sync定时扫描 →
  ├─ payload_hash = 相同 → 跳过（幂等）
  └─ payload_hash ≠ 相同 → 检查现有中台订单状态
      ├─ 已归档 → 新建订单
      ├─ 处理中(未发货) → 更新快照 + 重新预审 + 撤销旧通知
      ├─ 已部分发货 → 增量处理
      └─ 已取消 → 重新激活（已有 reactivate_cancelled_order）
```

## 四、一期核心自动化链路（修正版）

```
CRM审批完成
  │
  ├─ ① 订单类型识别（自动区分）
  │   ├─ 备货订单 → 跳过合同/回款审核，走备货预审规则
  │   └─ 销售订单 → 走标准预审规则
  │
  ├─ ② 前置条件检查（销售订单）
  │   ├─ CRM合同审批状态 = "已完成"？ → 否 → 阻断
  │   ├─ 销售邮箱？ → 否 → 通知补充
  │   └─ 附件齐全？ → 否 → 标记缺失
  │
  ├─ ③ 预审规则链
  │   ├─ 销售订单：11条标准规则（含合同金额/回款检查）
  │   └─ 备货订单：子集规则（跳过合同金额，价格取自产品价格表）
  │
  ├─ ④ ERP制单（自动）
  │   ├─ Save → Submit → Audit
  │   └─ 失败 → 异常
  │
  └─ ⑤ 发货通知 → OMS → 追踪 → 归档
```

**一期保留人工的环节：**
1. 借货/调仓的沟通（需要销售与亚马逊对接）
2. 一单多收货人的拆单确认（需B确认分拆方案）
3. 特殊需求的分发确认（需生产/物流确认）
