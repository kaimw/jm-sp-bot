# 1、项目立项目的及组织实施方式

## 项目名称

本项目名称为：基于企业邮箱与大模型的商务生产任务单智能体系统的研发。

## 项目目的

在企业商务订单协同领域，销售、商务、生产、财务及管理层之间仍大量依赖邮件完成订单需求传递、生产任务单下达、疑问反馈、补充确认、排产确认和周报统计等工作。目前，该流程主要依赖商务人员人工阅读邮件、复制粘贴订单信息、核对附件、整理生产任务单、跟踪生产反馈和人工汇总数据，存在响应延迟、信息遗漏、版本混乱、重复下达、异常追溯困难和统计口径不一致等问题。

随着公司业务规模扩大，订单邮件数量、附件类型和跨部门沟通频次持续增加，传统人工处理方式难以稳定满足生产协同的及时性、准确性和可追溯要求。尤其在订单信息不完整、生产部门提出疑问、销售补充答复、多轮沟通、订单变更或取消等场景中，若缺少统一的自动识别、规则初审、状态跟踪和审计机制，容易造成生产排产风险和管理统计失真。

本项目旨在研发一款面向商务生产任务单邮件流程的智能体系统。系统以企业邮箱为业务入口，结合邮件解析、附件解析、规则引擎、大模型辅助识别、工作流状态机、邮件自动生成与发送、异常分流、审计日志和周报统计等能力，实现销售订单需求从邮件接收、信息抽取、初审、任务单生成、生产反馈、销售补充、重新下达、生产确认到统计汇总的闭环处理。

通过本项目研发，可显著降低商务人员重复性操作工作量，提高订单处理响应速度，减少人工复制和邮件线程混乱造成的错误，形成可审计、可追踪、可统计的生产任务单数据资产，并为后续接入 ERP、MES、CRM、OA、企业微信、钉钉、飞书等系统奠定技术基础。

## 组织实施方式

本项目采用公司自主研发方式实施，由研发部牵头成立专项项目组，产品、后端、前端、测试、运维及业务代表协同参与。项目负责人负责研发计划制定、需求确认、技术方案评审、过程管理、风险跟踪、阶段验收和交付组织；产品人员负责业务流程梳理、需求定义、原型确认和验收标准制定；研发人员负责系统架构、后端服务、邮件适配、模型适配、工作流、管理台和部署方案开发；测试人员负责功能、接口、邮件链路、异常流程和回归测试；运维人员负责环境配置、数据备份、日志审计和上线支持。

项目采用阶段化研发与迭代验证相结合的方式推进。第一阶段完成需求调研、技术方案和可行性验证；第二阶段完成邮件接入、字段抽取、初审规则、任务单生成和管理台核心功能；第三阶段完成生产疑问、销售答复、重新下达、订单变更/取消、周报和运维能力；第四阶段开展真实邮箱端到端测试、灰度试运行、问题修复和正式交付。项目研发费用独立核算，阶段成果按计划评审，确保按期完成项目开发任务。

# 2、项目主要研究开发内容

本项目拟研发一套面向商务生产任务单邮件流程的智能体系统。系统通过腾讯企业邮箱 IMAP/SMTP 接入机器人邮箱，自动接收销售订单需求、生产反馈和销售补充邮件，解析邮件正文、HTML、原始 EML 和附件，识别邮件类型并抽取订单字段；通过可配置初审规则判断订单信息是否满足下达生产条件；通过模板引擎生成生产任务单、补充请求、生产疑问转发、销售补充回执、生产确认通知和周报邮件；通过状态机管理订单需求、生产任务单版本、疑问与答复、异常和关闭状态；通过管理台提供任务、邮件、外发、异常、周报、接入、生产邮箱、模板和运维管理能力。

系统当前研发内容主要包括：

1）企业邮箱接入与邮件适配：实现腾讯企业邮箱 IMAP 收件、SMTP 发件、邮件限频、发送失败重试、原始邮件保存、收件幂等和外发幂等。

2）邮件正文与附件解析：实现纯文本、HTML 正文清洗，支持 `.docx`、`.xlsx`、`.pdf`、`.zip`、`.txt`、`.csv` 附件解析，ZIP 默认支持最大 100MB、最多 1 层解压，并保存解析状态和错误信息。

3）邮件分类与业务识别：识别销售订单需求、生产疑问、销售答复、生产确认、订单变更、订单取消、非目标邮件和退信/自动回复等类型，并支持规则识别和大模型兜底分类。

4）订单字段抽取与证据留存：从邮件正文和附件中抽取客户名称、销售人员、产品/规格、数量、期望交期、订单号等关键字段，并记录字段来源证据、置信度、来源邮件和来源附件。

5）销售订单初审规则：支持配置初审开关、必填字段、自定义规则、风险关键词识别和初审失败原因，未通过初审时自动生成补充信息请求邮件，并进入异常队列。

6）生产任务单生成与版本管理：基于模板生成生产任务单主题和正文，生成任务编号和版本号，支持销售补充后生成新版任务单并重新下达生产。

7）生产疑问与销售答复闭环：生产部门提出疑问后，系统自动转发销售；销售答复后，系统判断答复完整性，更新结构化字段，生成新版任务单，并向销售发送重新下达成功回执。

8）生产确认与任务关闭：生产部门确认排产后，系统自动记录确认结果，关闭任务，生成生产确认通知，并纳入周报统计。

9）订单变更与取消分流：识别订单变更和取消申请，未确认排产前可生成变更版任务单，已确认排产后的变更和取消必须转人工异常处理。

10）异常队列与人工处理：对缺字段、路由缺失、附件解析失败、任务关联失败、销售答复不充分、沟通轮数超限、订单取消和已排产变更等场景创建异常，支持人工补字段恢复流程或关闭异常。

11）周报与统计：按北京时间统计本周、本月、本年需求总数、已确认数、未确认数、产品维度订单统计和销售 Top10，支持周报预览、邮件入队、PDF 导出和 CSV 导出。

12）管理台和运维能力：提供 Web 管理台，支持登录认证、工作台指标、任务查询、邮件查询、外发管理、初审规则、生产邮箱、模板、接入配置、附件解析、清理预览、备份和审计日志。

## 2.1 项目拟采用核心技术

1）企业邮箱 IMAP/SMTP 适配技术。系统通过腾讯企业邮箱 IMAP 协议同步机器人邮箱中的新邮件，通过 SMTP 协议发送自动生成的业务邮件。邮件接入模块实现收件幂等、原始 EML 保存、附件保存、发送队列、失败重试和账号限频，保证邮件链路的稳定性和可追溯性。

2）邮件解析与附件内容抽取技术。系统基于 Python 邮件解析能力处理邮件头、发件人、收件人、主题、纯文本正文、HTML 正文和原始 EML；基于 `python-docx` 解析 Word 段落和表格，基于 `openpyxl` 解析 Excel 表格，基于 `pypdf` 提取可解析文本 PDF，基于 ZIP 安全解包策略解析压缩包内支持格式文件。

3）规则分类与大模型兜底识别技术。系统优先采用确定性关键词、邮件发件人、任务编号、订单编号和邮件线程关系进行分类；对规则无法稳定识别的非目标或低置信邮件，可调用 OpenAI 兼容模型服务进行兜底分类和字段抽取，提升自然语言邮件和非标准格式邮件的处理能力。

4）订单字段抽取与证据链技术。系统将邮件正文和附件解析文本合并为可抽取语料，识别客户、产品、数量、交期、订单号和销售人员等字段，并为每个字段记录来源类型、来源邮件、来源附件、证据文本和置信度，便于商务人员复核和后续审计。

5）可配置初审规则引擎。系统通过后台配置必填字段和自定义规则，支持字段级和全文级判断，覆盖缺字段、风险关键词、格式异常和业务约束场景。规则执行结果会写入订单需求状态、异常详情和审计日志。

6）工作流状态机与任务版本管理技术。系统以生产任务为核心对象，管理从订单需求识别、初审、任务草稿、任务下达、生产疑问、销售答复、重新下达、生产确认、变更/取消、异常处理到关闭的状态流转。每次重新下达都会生成新版本任务单，避免邮件版本混乱。

7）模板渲染与外发邮件队列技术。系统采用模板变量渲染任务单、补充请求、生产疑问、销售回执、生产确认和周报邮件。所有外发邮件先写入外发队列，发送成功后回写状态；通过业务幂等键防止重复下达和重复发送。

8）审计、清理和备份技术。系统对关键动作记录审计事件，支持过期非目标邮件和临时记录清理预览与执行，支持 SQLite 数据和附件目录 ZIP 备份，保证研发样机具备基本运维能力。

9）模块化单体架构与容器化部署技术。系统采用 Python FastAPI 后端、SQLAlchemy 数据访问、本地静态管理台、SQLite 默认存储和 Docker Compose 部署方式，既便于 MVP 快速交付，也为后续替换 PostgreSQL、对象存储、Redis/Celery 和前端工程化框架保留扩展空间。

## 2.2 技术指标

1）邮件接入能力：支持腾讯企业邮箱 IMAP/SMTP 接入；支持机器人邮箱、IMAP/SMTP 主机、端口、显示名、签名、邮箱限频和 worker 周期运行期配置；支持自动同步和手动同步。

2）邮件分类能力：支持 `SalesOrderRequirement`、`ProductionQuestion`、`SalesClarificationReply`、`ProductionScheduleConfirmation`、`OrderChangeRequest`、`OrderCancelRequest`、`NonTarget`、`BounceOrAutoReply` 等主要类型识别；MVP 阶段订单需求邮件识别准确率目标不低于 90%。

3）字段抽取能力：支持从邮件正文和附件解析文本中抽取客户名称、产品/规格、数量、期望交期、订单号、销售人员和销售邮箱；MVP 阶段关键字段抽取准确率目标不低于 90%。

4）附件解析能力：支持 `.docx`、`.xlsx`、`.pdf`、`.zip`、`.txt`、`.csv`；ZIP 默认最大 100MB，最大解压层级 1 层；不支持格式保存元数据并标记跳过，不执行附件内任何文件。

5）初审规则能力：支持初审开关、必填字段配置、自定义规则配置、风险关键词识别、初审失败原因生成和补充请求邮件自动生成；初审通过场景可生成生产任务单。

6）任务单生成能力：从收到销售邮件到生成生产任务单或补充请求的目标耗时不超过 2 分钟；任务单主题必须包含任务编号和版本号，确保后续邮件可追溯。

7）生产反馈处理能力：生产部门提出疑问后，系统应在 5 分钟内转发销售或进入异常队列；生产确认排产后可自动关闭任务并生成通知。

8）销售补充闭环能力：支持销售补充缺失字段、答复生产疑问、触发新版任务单生成和重新下达；支持最大沟通轮数限制，默认 3 轮，超过后关闭并要求重新发起完整订单。

9）外发稳定性：外发邮件采用队列处理和业务幂等键，支持失败重试；目标邮件发送成功率不低于 99.5%，重复下达率关键场景为 0。

10）管理台能力：提供工作台、任务、模拟订单、初审规则、邮件、外发、异常、周报、接入、生产邮箱、模板和运维 12 类功能入口。

11）统计与报表能力：支持本周、本月、本年三个周期统计，支持需求总数、已确认数、未确认数、产品维度统计、销售 Top10、周报邮件、PDF 和 CSV 导出。

12）安全与运维能力：敏感配置不在页面回显；有效订单邮件默认长期保留；非目标邮件和临时记录默认保留 30 天；默认存储预算 10GB；支持清理预览、执行清理、立即备份和审计查询。

## 2.3 创新点

1）以企业邮箱为入口的订单智能体闭环。本项目不要求业务人员改变现有邮件协同习惯，而是在原有企业邮箱流程上叠加智能识别、自动抽取、任务生成和状态跟踪能力，降低系统上线阻力。

2）规则引擎与大模型协同的业务识别方式。系统优先使用确定性规则保障可控性和可解释性，对非标准邮件再使用大模型兜底识别和抽取，实现效率与风险控制的平衡。

3）字段证据链与任务版本管理。系统不仅抽取订单字段，还记录字段来源邮件、来源附件和证据文本；销售补充后自动生成新版任务单，解决传统邮件流程中“哪个版本是最新版本”的管理痛点。

4）生产疑问与销售答复自动闭环。系统将生产部门疑问自动整理转发销售，销售答复后重新下达生产，实现跨部门沟通的自动跟踪和闭环，减少商务人员人工转发与催办。

5）业务异常可恢复机制。缺字段、路由缺失、关联失败、答复不充分、订单取消和已排产变更等场景统一进入异常队列，支持人工补字段恢复流程或关闭异常，避免自动化系统在高风险场景中误处理。

6）面向管理决策的自动周报。系统基于任务状态自动生成本周、本月、本年统计，覆盖产品、销售、确认和未确认订单等维度，减少人工统计时间并统一经营口径。

# 3、与同类产品的竞争优势

1）更贴近企业现有业务流程。通用邮件客户端、普通工单系统或 OA 流程系统通常需要业务人员主动填写表单或迁移工作习惯，本项目直接接入企业邮箱，以邮件为业务入口，对销售和生产部门影响小，落地成本低。

2）具备完整的生产任务单业务闭环。普通邮件自动化工具多停留在分类、通知或模板发送层面，本项目覆盖订单需求、初审、任务单、生产疑问、销售答复、重新下达、生产确认、变更/取消和周报统计，业务链路更完整。

3）风险控制更强。系统通过初审规则、风险关键词、路由校验、业务幂等、异常队列、人工补字段和最大沟通轮数限制，避免在订单取消、已排产变更、字段缺失和答复不充分等高风险场景中无人值守处理。

4）可审计和可追溯能力更强。系统保存原始邮件、附件、字段证据、外发任务、任务版本、疑问答复、异常详情和审计事件，能够满足商务复盘、生产追溯、管理统计和责任确认要求。

5）具备模型服务可替换能力。系统采用 OpenAI 兼容模型 Provider 设计，支持运行期配置 API Base、模型名称和密钥，后续可切换 Dify、DeepSeek、私有模型或其他兼容模型，降低供应商锁定风险。

6）运维与扩展成本低。MVP 采用模块化单体、FastAPI、SQLite、Docker Compose 和本地静态管理台，便于快速部署和验证；后续可平滑扩展到 PostgreSQL、对象存储、异步队列和企业级前端工程。

# 4、项目的实施计划以及经费概算

## 4.1 工作内容

1）项目启动：收集销售、商务、生产、财务和管理层对生产任务单邮件流程的实际需求，梳理现有邮件处理痛点，完成可行性分析、研发目标、项目边界和人员安排，召开项目启动会并形成立项报告。

2）需求定义：基于订单需求邮件、生产任务单模板、生产反馈邮件、销售补充邮件和周报口径，形成 PRD、技术方案、数据库与工作流设计、验收指标和测试计划。

3）总体设计：完成系统总体架构、模块边界、数据模型、状态机、邮件分类规则、字段抽取策略、初审规则、模板变量、异常类型、外发策略、清理和备份策略设计。

4）邮件接入研发：实现腾讯企业邮箱 IMAP/SMTP 适配、原始邮件保存、附件保存、收件幂等、外发队列、发送失败记录、邮箱限频和自动 worker。

5）解析与识别研发：实现正文清洗、附件解析、邮件分类、字段抽取、大模型兜底分类和字段证据记录。

6）业务工作流研发：实现销售订单初审、补充请求、生产任务单生成、生产疑问转销售、销售答复后新版任务单、生产确认关闭、订单变更/取消异常分流和最大沟通轮数控制。

7）管理台研发：实现登录、工作台、任务、模拟订单、初审规则、邮件、外发、异常、周报、接入、生产邮箱、模板和运维页面。

8）报表与运维研发：实现周报预览、周报邮件、PDF/CSV 导出、附件解析记录、存储占用、清理预览、执行清理、立即备份和审计日志。

9）测试执行：开展单元测试、接口测试、邮件端到端测试、附件解析测试、异常流程测试、外发重试测试、周报统计一致性测试和回归测试。

10）灰度试运行：使用测试邮箱和少量真实业务邮件进行灰度验证，收集销售、商务、生产反馈，优化关键词、初审规则、模板和异常处理方式。

11）内部验收：依据立项指标、PRD 和测试报告进行内部验收，确认系统在功能、稳定性、安全性、可追溯性和报表口径方面达到项目目标。

12）上线与维护：整理部署文档、用户手册、测试报告和验收资料，完成系统上线、数据备份、运维交接和后续迭代计划。
