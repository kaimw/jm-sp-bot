## 一、内测目的

本系统的数据源头是**纷享销客CRM**中的审批完成订单。CRM负责人在内测中扮演最关键的角色之一：**验证CRM数据能否被中台正确读取、解析并转化为可处理的订单**。

**您需要重点关注的问题**：

> 💡 CRM里的订单，进到中台后准不准？
> 💡 哪些订单应该进来、哪些不应该进来，有没有控制好？
> 💡 客户名称、SKU编码、金额，有没有对上主数据？
> 💡 如果CRM订单变更或撤销，中台是怎么反应的？

**一期完整链路说明**：

```
CRM（纷享销客）审批完成订单
          ↓ 系统自动同步
中台建单（MP-前缀订单号）
          ↓
字段标准化 + 主数据匹配
（客户 / SKU / 仓库 / 金额）
          ↓
8条规则自动预审
     ↙          ↘
  通过              失败
   ↓                 ↓
生成发货通知草稿    异常工单
   ↓                 ↓
人工确认           AI诊断 + 人工处理
   ↓
金蝶ERP自动制单
   ↓
发货通知邮件 → 物流手动在吉客云操作发货
```

> [!IMPORTANT]
> **一期边界**：订单闭环到「ERP制单完成 + 发货通知邮件发出」为止。OMS（吉客云）API自动对接、发货状态自动回写是二期计划，本次内测不涉及。

## 五、内测流程与操作步骤

### ✅ 任务一：确认CRM同步配置（5分钟）

**目的**：了解哪些CRM订单会进入中台，确认范围配置正确。

**操作步骤**：

1. 进入「🔗 接入配置」→「CRM配置」页面
2. 确认以下配置项的状态：

   | 配置项 | 含义 | 期望值 |
   |--------|------|-------|
   | CRM自动同步开关 | 是否启用自动拉取 | 应为「开启」 |
   | 一期范围过滤开关 | 是否只处理特定范围订单 | 内测期建议开启 |
   | 一期范围配置 | 指定哪些业务线/部门/审批状态进入中台 | 确认与预期一致 |

3. 查看「一期范围配置」JSON内容，确认：
   - `approved_values`（审批状态值）是否包含您CRM中的审批完成状态值
   - `include_owner_departments`（部门范围）是否包含参与内测的部门

**检查要点**：
- [ ] CRM同步开关已开启
- [ ] 范围配置中的审批状态值与CRM实际状态名称一致
- [ ] 部门范围配置正确，不会漏掉参与内测的销售

> [!WARNING]
> 如果审批状态值配置错误（比如CRM里叫"审批通过"但配置写的是"approved"），会导致所有CRM订单都进不来。这是最常见的配置错误，请重点核查。

### ✅ 任务三：验证范围外订单的过滤效果（5分钟）

**目的**：确认不在一期范围内的CRM订单不会错误地进入中台。

**操作步骤**：

1. 在CRM中找一条**不在内测范围内**的订单（如其他部门或其他业务线）
2. 确认该订单的CRM审批状态
3. 回到中台「📦 订单处理」，搜索该订单
4. **预期结果**：该订单不应出现在中台订单列表中

**检查要点**：
- [ ] 范围外订单未进入中台（确认过滤有效）
- [ ] 范围内订单全部进入中台（确认无漏掉）

### ✅ 任务四：查看CRM数据问题导致的异常（10分钟）⭐ 重点任务

**目的**：当CRM订单数据有缺陷时（如客户主数据匹配不上、SKU未映射），验证中台的异常提示是否清晰。

**操作步骤**：

1. 点击左侧「🔍 异常」
2. 筛选以下类型的异常，逐一查看：

   | 异常类型 | 含义 | 您需要关注的点 |
   |---------|------|--------------|
   | 客户映射失败 | CRM的客户名称在中台主数据中找不到对应 | 提示的客户名称是否就是CRM中的原始名称？映射建议是否合理？ |
   | SKU映射失败 | CRM的产品编码在中台没有对应标准SKU | 提示是否清楚说明是哪个SKU匹配不上？ |
   | 字段缺失阻断 | 收货人信息、地址等关键字段为空 | 提示的缺失字段说明是否准确？ |

3. 对于每种异常类型，点击查看详情，检查：
   - 异常描述能否让您快速定位CRM数据问题的根源？
   - AI诊断建议是否可操作（比如"建议在主数据中添加该客户的映射关系"）？
   - 是否显示了来自CRM的原始数据快照（便于核查）？

4. 尝试使用「**补字段**」功能，对缺少收货地址的异常：
   - 手动补充完整的收货地址
   - 确认系统能否根据补充信息重新通过预审

**检查要点**：
- [ ] 异常描述准确指向CRM数据问题
- [ ] AI诊断建议与实际业务情况吻合
- [ ] 补字段功能可用，补充后能触发重新预审
- [ ] 操作记录有审计（便于追溯谁做了什么修改）

### ✅ 任务五：验证客户主数据映射（10分钟）⭐ 重点任务

**目的**：CRM中的客户名称需要映射到中台/OMS的客户编码，这是最容易出问题的地方。

**背景知识**：
> CRM中的客户可能叫「北京ABC贸易有限公司」，但OMS（吉客云）里的客户编码是「CUSTOMER_001」。系统需要通过主数据映射表把两者对应起来。

**操作步骤**：

1. 进入「主数据」→「客户映射」页面
2. 找到您内测所用测试订单的客户，确认：
   - CRM客户名称/ID 是否在映射表中有对应条目
   - 映射的客户编码是否正确

3. 故意使用一个**未配置映射的客户**下单（通过异常测试场景），观察：
   - 系统是否产生「客户映射失败」异常
   - 异常提示是否清楚说明哪个客户需要配置映射

4. 如果有条件，尝试在主数据中**新增一条客户映射**：
   - 添加测试客户的CRM ID 和 OMS编码的映射关系
   - 对之前失败的订单触发重新处理
   - 验证重新处理后是否通过

**检查要点**：
- [ ] 客户映射配置界面操作流畅
- [ ] 未映射客户会产生明确的异常提示
- [ ] 新增映射后，相关订单能重新通过预审
- [ ] 是否存在CRM有、但主数据里没有的客户？（请记录客户列表）

### ✅ 任务六：测试CRM订单变更场景（10分钟）

**目的**：当CRM订单审批后被销售修改（比如改了数量或地址），中台是否能正确响应。

**操作步骤**：

1. 选取一条**已进入中台且处于「发货通知待确认」状态**的测试订单
2. 在CRM中修改该订单的某个字段（比如数量 +1 或修改收货地址）
3. 等待系统自动同步（或手动触发同步），回到中台查看：
   - 该订单状态是否变为「CRM变更待处理」或产生高优先级异常
   - 原有的发货通知草稿是否被作废（防止用旧数据发货）
4. 查看系统生成的**变更异常详情**，确认：
   - 变更前后的数据对比是否显示清楚
   - 操作建议是否合理（比如「请重新确认发货通知」）

**检查要点**：
- [ ] CRM变更后中台能感知到变化
- [ ] 原发货通知草稿自动失效，防止误发货
- [ ] 变更异常的数据对比清晰（变更前 vs 变更后）
- [ ] 提示操作建议，引导人工重新确认

> [!WARNING]
> 这是一个高风险场景！如果CRM订单改了但中台发货通知没有更新，就会按旧数据发货。请认真验证此场景。

### ✅ 任务七：验证预审规则与CRM数据的匹配性（10分钟）

**目的**：从CRM数据视角评估预审规则是否合理，有无误拦截或漏放的情况。

**操作步骤**：

1. 进入「⚙️ 初审规则」，查看当前配置的 8 条预审规则：

   | 规则类型 | 说明 |
   |---------|------|
   | 字段完整性检查 | 必填字段是否都有值 |
   | SKU有效性检查 | SKU必须在主数据中存在 |
   | 客户主数据匹配 | 客户必须能映射到标准编码 |
   | 金额合理性检查 | 金额不为零，与CRM一致 |
   | 收货地址精度检查 | 地址必须精确到街道/门牌号 |
   | 库存预判检查 | 订单数量不超过可用库存 |
   | 发货条件检查 | 必要的发货信息完整 |
   | 业务范围检查 | 订单属于一期纳入范围 |

2. 对照您的测试订单，思考并反馈：
   - 是否有规则会误拦截**本应通过**的正常订单？（请举例说明）
   - 是否有规则会漏放**本不应通过**的问题订单？（请举例说明）
   - 是否有您认为应该增加的规则？（请描述业务场景）

**检查要点**：
- [ ] 所有合法测试订单均通过预审
- [ ] 故意制造的问题订单均被正确拦截
- [ ] 规则描述语言通俗易懂，无歧义
- [ ] 预审结果的说明能让商务/销售看懂

## 八、常见问题 Q&A

**Q：为什么CRM里审批通过的订单，在中台看不到？**
> A：可能原因：①同步还未触发（等待或手动触发）②范围配置不包含该订单类型 ③CRM的审批状态值与配置不匹配。请按顺序排查。

**Q：中台显示的客户名称和CRM不一样？**
> A：如果客户名称有细微差异（如全称vs简称），会导致主数据匹配失败。请在主数据中配置别名或补充映射关系。

**Q：SKU映射失败，但我在主数据里确认有这个SKU？**
> A：可能是渠道SKU与标准SKU的对应关系未配置。请在「主数据-渠道SKU映射」中检查。

**Q：CRM订单的合同PDF附件，中台能看到但内容解析不出来？**
> A：PDF扫描件（图片型）当前无法自动解析文字。如果是扫描版合同，系统会保留文件但无法提取字段，需要人工录入关键信息。

**Q：测试CRM变更场景时，发现中台没有响应变更？**
> A：可能是同步周期问题（系统默认60秒同步一次）。请等待同步周期后再观察，或联系管理员手动触发同步。
