# CFO讲话深度分析 — 对中台设计的核心影响

## 一、CFO的原话逐句拆解

> **"吉客云只有一个主体，发货涉及到内部不同的公司，影响各公司收入确认和存货归属的准确性。现在海外对账太麻烦了。"**

**含义：** 吉客云（OMS/WMS）只能配置一个公司主体。发出去的货到底是从哪个主体的库存出的、收入该算给谁——吉客云的数据无法回答这个问题。导致海外对账非常困难。

> **"主要是订单主体，发货主体，和调货的问题。而且涉及跨国了。"**

**含义：** 三个核心问题：①这笔订单是谁签的（订单主体）②这批货从哪个主体的仓库出的（发货主体）③如果从A主体调到B主体，中间的调货关系怎么处理。而且这不是国内不同公司之间的调拨——**涉及跨国，有税务/海关/转移定价的法律要求。**

> **"每一个公司都是一个独立的税务主体，独立的财务核算。"**

**含义：** 深圳积木易搭、武汉尺子科技、积木易搭（香港）——每一个都是**独立的法人实体**，各交各的税，各做各的账。不能混在一起算。

> **"吉客云如果按公司主体来，是最清楚的，但是物流的工作量会增加很多。"**

**含义：** 如果每个主体单独用一套吉客云，账目一清二楚。但物流（仓库发货）要同时登录多个系统操作，工作量剧增。所以**吉客云合在一起用（一个主体）是物流效率的妥协**，但牺牲了财务清晰度。

> **"如果在一个系统里，就必须每个公司每笔流向非常清楚，从订单，仓库（发货，还是调其他公司的货），到收款。"**

**含义：** 这是对中台的核心要求——既然吉客云不分主体，那中台就必须**把每一笔货的流向记录清楚**：这个单是谁的、货从哪个主体的库存出的、收款归谁。**中台要成为"跨主体追踪的真相来源"。**

> **"而且因为我们的业务特性，导致信息滞后。调其他公司的货就属于跨国销售了。但之前并没有人走流程，财务做到最后才发现再去调。"**

**含义：** 现实中的问题——深圳没货就去香港的亚马逊仓库调了，但**没有任何流程记录这笔调货**。财务做账做到最后发现库存对不上、收入该算给谁不清楚。**信息滞后是问题的根源，不是调货这个动作本身。**

> **"现在有海外仓管，已经好很多，逐步做细做准。"**

**含义：** 已经有改善（有海外仓管了），但还不够。CFO希望系统能进一步做细做准。

## 二、核心结论：中台要解决什么

### 2.1 吉客云的局限性

```
┌───────────────────────────────────────────────┐
│                  吉客云 (OMS)                   │
│             只支持一个主体                       │
│   能看到：货从哪发出、发到哪、物流单号          │
│   看不到：这批货是哪个主体的库存、              │
│           收入该算给深圳还是香港                │
└───────────────────────────────────────────────┘
         ↑ 中台要补充的信息层 ↓
┌───────────────────────────────────────────────┐
│             中台 (商务AI Agent)                   │
│   补充记录：                                     │
│   · 订单主体（深圳签的）                         │
│   · 发货主体（从香港的库存出的）                 │
│   · 调货记录（深圳→香港的转移）                 │
│   · 结算依据（内部调拨价格）                     │
│   · 收款归属（钱该进谁的账）                    │
└───────────────────────────────────────────────┘
```

### 2.2 中台设计的核心原则

从CFO的讲话可以提炼出四条设计原则：

**原则1：每笔货要有"主体轨迹"**
不只是"货从A仓到B仓"，还要记录"货从香港主体的库存出、给深圳主体的客户发"。

**原则2：调货即销售（跨国的）**
深圳调香港的货 = 深圳向香港采购 + 香港向深圳销售。这是两个独立税务主体之间的交易，必须记录清楚。

**原则3：信息不能滞后**
财务不是事后才发现——系统要在调货发生的那一刻就生成记录，推送给相关方。

**原则4：不给物流增加额外负担**
吉客云不分主体是物流效率的考量（否则物流要操作多套系统）。中台的设计不能要求物流改变操作方式——**中台在后台记录主体流向，不影响物流的实际操作。**

## 三、重新理解"调货"在中台中的定位

之前我一直以为跨主体调货是**二期功能**，需要等ERP接口、等财务确认规则。但现在理解到：

**跨主体调货的记录是中台的"主线任务"，不是附加功能。**

因为CFO的核心诉求不是"自动调货"——而是"清晰地记录每一笔货的流向"。即使一期不做自动拆单、不做自动结算，也**必须能够回答以下问题**：

```
订单MP-001：
  ├→ 下单主体：深圳（CRM销售合同）
  ├→ 出货明细：
  │   ├→ 物料A*50台 → 深圳仓发（深圳主体的库存）
  │   └→ 物料B*50台 → 香港欧洲仓发（香港主体的库存）
  ├→ 调货记录：
  │   └→ 深圳向香港调货：物料B*50台
  ├→ 结算信息：
  │   ├→ 外部：客户付给深圳
  │   └→ 内部：深圳应付给香港（调拨价）
  └→ 状态：已通知深圳财务 + 香港财务
```

### 3.1 一期的最低要求

一期即使不自动执行调货，中台也需要：

| 能力 | 一期实现 | 说明 |
|------|---------|------|
| 记录订单主体 | 从CRM同步 | CRM已有客户/渠道信息 |
| 识别发货主体 | 库存表中有仓库归属 | Excel导入时标记仓库属于哪个主体 |
| 标记跨主体订单 | 新增字段 fulfillment_entity | 订单部分物料从非下单主体的库存出 |
| 生成调货记录 | 新增 InterEntityTransfer 表 | 记录"谁→谁，什么物料，多少" |
| 通知财务 | 复用现有通知机制 | 订单确认后通知对应财务人员 |

### 3.2 不做和不做的原因

| 不做 | 原因 |
|-----|------|
| 不自动拆ERP单 | 需要金蝶多组织确认，一期时间不够 |
| 不自动结算 | 调拨价格规则需财务确认 |
| 不自动税务处理 | 跨国税务太复杂，需专项设计 |

## 四、总结：中台 vs 吉客云 vs 金蝶的角色划分

```
┌──────────────────────────────────────────────────────────────┐
│                        中台 (AI Agent)                        │
│  职责：跨主体追踪 + 订单类型的识别 + 预审 + ERP制单 + 发货通知 │
│  记录：每个订单的主体流向、调货记录、通知干系人               │
│  约束：必须清楚记录"哪个主体的订单→用了哪个主体的库存→     │
│        哪个主体发货→收入归哪个主体"                          │
└──────────────────────────────────────────────────────────────┘
         ↑ 收到发货指令 ↓                   ↑ 制单指令 ↓
┌───────────────┐              ┌───────────────────────────────┐
│   吉客云(OMS)   │              │      金蝶云星空(ERP)          │
│  只认一个主体   │              │      区分多个主体             │
│  只管发货执行   │              │      FSaleOrgId区分组织       │
└───────────────┘              └───────────────────────────────┘
```

**核心矛盾：** 吉客云只支持一个主体，但实际业务涉及多个独立税务主体。
**中台的解决方案：** 不在吉客云层面解决（改不了），而是在中台层面补充"主体流向记录"，给财务提供审计线索。
