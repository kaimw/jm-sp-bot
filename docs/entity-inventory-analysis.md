# 主体-调货-预审 — 需求深度分析（根据最新对话修正）

> 对话时间：2026-06-27（第三段讨论）

---

## 一、我的理解错误：之前以为"仓库决定主体"

**这是错的。**

| 我之前以为 | 对话中的真实情况 |
|-----------|----------------|
| 仓库归属决定用哪个主体做单 | **PI决定主体，仓库只是发货位置** |
| 调货是因为"仓库主体≠销售主体" | 调货是因为**库存不够**，不是为了统一主体 |
| 借货（从亚马逊调）需要换主体 | **借货不换主体**，还是原主体做单 |
| 改成国内仓发=只是换个仓库 | 改成国内仓发=**主体从香港变深圳**（因为跨国了） |

## 二、真实流程拆解

### 2.1 库存够的流程（最简单）

```
销售发邮件：PI是香港公司，从欧洲仓发
  → B查涛哥的库存表：欧洲仓有货
  → 香港账套做单，备注"欧洲仓发货"
  → 发货主体=香港，与销售PI一致
```

**核心：PI决定主体，仓库只是一个备注。**

### 2.2 库存不够的流程（调货决策树）

```
销售发邮件：PI是香港公司，从欧洲仓发
  → B查库存表：欧洲仓没货（或不够）
  → B问涛哥：有在途吗？什么时候到？
  → B反馈销售，确认方案：

  ┌── 方案A：等货
  │     → 等涛哥在途到仓
  │     → 主体不变（香港），发货时间推迟
  │
  ├── 方案B：借货（从亚马逊借）
  │     → 亚马逊有备货，先用了
  │     → 主体不变（香港），仓库改成"亚马逊仓"
  │     → 不还（亚马逊不够了继续备）
  │     → T问"借货要还吗？" B答"好像没还过"
  │
  ├── 方案C：改从国内仓发
  │     → 主体变化！香港→深圳
  │     → 深圳积木账套做单
  │     → 客户：香港公司
  │     → 备注：终端客户名称
  │     → 这是真正的"跨主体"场景
  │
  └── 方案D：取消
        → 取消订单
```

### 2.3 关键理解：什么情况下主体会变？

**只有一种情况：从海外仓发改为国内仓发。**

```
PI是香港的（香港主体）
  → 从欧洲仓发 → 香港账套做单 ✅（主体与PI一致）
  → 欧洲仓没货 → 改从亚马逊仓发（借货）
     → 还是香港账套做单 ✅（借货不改变主体）
  → 亚马逊也没货 → 改从国内（武汉仓）发
     → 变成深圳账套做单 ⚠️ 主体变了！
     → 客户名称：香港公司
     → 备注：终端客户名称
```

**为什么改国内仓发会导致主体变？** 因为这是**跨国家/地区的出货**——国内仓的货属于深圳公司，深圳公司"卖"给香港公司，香港公司再卖给终端客户。这触及了CFO说的"跨国销售"和"独立税务主体"。

## 三、"借货"到底是什么？

对话澄清了一个重要概念：

```
借货（从亚马逊借）：
  ├→ 场景：欧洲仓没货，亚马逊仓有
  ├→ 操作：香港账套做单，发货仓库改成"亚马逊仓"
  ├→ 主体：不变，还是香港
  ├→ 本质：用了亚马逊的备货库存
  └→ 还吗？B说"好像没还过"，就是先用，亚马逊不够了继续备

对比 调货（改国内仓发）：
  ├→ 场景：海外仓完全没货，改成从国内发
  ├→ 操作：深圳账套做单
  ├→ 主体：从香港变成深圳
  ├→ 本质：深圳把货卖给香港，香港再卖给终端
  └→ 需要内部结算
```

**所以"借货"和"调货"是两回事：**
- **借货**：同一主体内，换仓库发货。不涉及主体变更，不需要内部结算
- **调货**：跨主体之间的货权转移。涉及主体变更，需要内部结算

## 四、仓库不区分主体

B明确说：**"仓库不区分主体"**

这意味着：
- 欧洲仓的货架上，放着香港公司的货和深圳公司的货，**混在一起**
- 发货时，涛哥（仓管）只负责从货架上拿货发出，**不关心这批货归哪个主体**
- **主体是谁，由做单时的PI决定，不是由仓库决定**

这对中台设计的含义：
- 库存表（涛哥给的Excel）是按**仓库**统计的，不是按**主体**统计的
- 所以库存检查时，中台只能查"欧洲仓有多少货"，不能查"欧洲仓里属于香港的有多少"
- 因为**仓库本身不区分**，所以**中台也不需要区分**

## 五、AI"内部调货单"要做什么？

T提出：**"调仓的结果就会产生主体不一致，再然后AI就做一个内部调货单"**

这个"内部调货单"的本质是：

```
当发生跨主体调货时（海外改国内发）：
  → AI自动生成内部调货单
  → 记录：
      ├─ 调出主体：深圳（从国内仓发的货归属深圳）
      ├─ 调入主体：香港（PI上的销售主体）
      ├─ 物料+数量：调了什么
      ├─ 调拨价格：需要财务确认
      ├─ 关联的CRM订单
      └─ 通知：深圳财务 + 香港财务
```

**这个调货单不是ERP调拨单**（不走金蝶），它是中台内部的**记录单**，给财务对账用的。

## 六、对中台设计的核心影响

### 6.1 库存检查的流程

```
中台收到订单后 → 判断订单主体（从PI/CRM）
  → 根据订单的发货要求（附件/PI中指定仓库）
  → 查对应仓库的库存（Excel导入表）
  ├─ 够 → 正常流程（ERP做单，备注发货仓）
  └─ 不够 → 生成异常任务
      └─ "欧洲仓库存不足，请确认方案：等货/借货/改国内/取消"
      └─ B确认方案后：
          ├─ 借货 → 同一个主体，换仓库备注
          └─ 改国内 → 主体变更 + AI生成内部调货单
```

### 6.2 中台需要的字段

```
CRM/PI解析：
  ├─ sales_entity：销售主体（从PI/CRM解析）
  ├─ warehouse：指定发货仓（从邮件/附件解析）
  └─ warehouse_entity：仓库所属主体（从配置表映射）

库存表（Excel导入）：
  ├─ warehouse：仓库名称
  ├─ material_code：物料编码
  ├─ quantity：库存量
  └─ 不需要"主体"字段（因为仓库不区分主体）

内部调货单（InterEntityTransfer）：
  ├─ source_entity：调出主体
  ├─ target_entity：调入主体
  ├─ material：物料+数量
  ├─ reason：调货原因（库存不足）
  ├─ order_id：关联中台订单
  ├─ status：待确认/已确认
  └─ generated_by：AI自动生成
```
