## 一、多公司主体之间的调货

### 现状：一个订单涉及多个公司主体

原文 [17:42-22:09] B 演示了一个典型场景：

```
CRM 下单 → 销售主体：香港公司
         → 出库账号：深圳账号（从国内发）
         → 仓库：武汉仓发货
         → PI 开给：香港公司
```

即：**销售主体(香港公司) ≠ 金蝶出库账号(深圳账号) ≠ 发货仓库(武汉仓/欧洲仓)**

### 调货场景（多个主体间的物料迁移）

原文 [20:59-22:09] B 演示了完整的调货案例：

1. 原始订单：国内发，全部从武汉仓出
2. 发现部分物料国内没货但欧洲仓有 → 沟通销售确认
3. 结果：12台从国内发改为欧洲仓发货
4. 操作：**在香港账号另做一个新单**（因为欧洲仓出对应香港账号）
5. 原单同步更新：去掉这12台，其他物料还是武汉仓发

### 多主体原则（从对话中提炼）

| 原则 | 原文依据 |
|------|---------|
| 一个金蝶账号对应一个公司主体 | B多次切换「深圳账号」「香港账号」操作 |
| 仓库归属决定用哪个账号出单 | 欧洲仓→香港账号，武汉仓→深圳账号 |
| PI（销售合同）主体决定客户归属 | 香港PI→香港公司客户 |
| 调货后原单和新单要同步更新 | 原单去掉12台，新单加12台 |

### 一期处理建议

调研中 T 没有深入讨论多主体问题，一期重点是睿数（单主体）流程。但数据模型需要预留：
- `CrmSalesOrder` 已有 `settlement_method` 字段
- ERP 制单时需要能指定**金蝶账号/组织**（`FSaleOrgId`）

## 二、拆单、改单的条件和判断原则

### 改单的 5 种类型

原文 [51:04-51:28] B 列举：

| 类型 | B的原话 | 含义 |
|------|--------|------|
| ① 取消 | "原来的这个不要了" | 整单取消 |
| ② 加量 | "多加几个" | 原物料数量增加 |
| ③ 改量 | "一个改成两个" | 物料数量变更 |
| ④ 加新物料 | "多加几个物料" | 新增SKU |
| ⑤ 减物料 | 承接上文 | 删除部分物料 |

### 改单触发方式

原文 [51:39-53:46]：

```
销售在CRM修改 → 通知B → B告诉销售CRM要同步调整
              → 销售更新CRM（含附件/PI）
              → CRM订单编号不变，内容变
```

### 改单的判断原则（核心）

**原则1：已闭环的订单 → 建议新建**
原文 [50:50-50:54]
T："如果修改订单，状态已经闭环了……"
B："那形成了新的订单是不是？"
T："对我来说其实新的订单是最方便的"

**原则2：未闭环的订单 → 在原单上叠加修改**
原文 [51:04-51:28]
B："修改有很多种情况，只能在原单上叠加，或者新增创建订单"

**原则3：时间间隔太久的补需求 → 新增订单，不在原单改**
原文 [24:08-24:35]
B："以前我们在原单上直接改……但现在张燕说时间间隔太久又要补需求的话，就新增一个订单"

**原则4：已部分发货的 → 区分已发和未发**
原文 [01:04:30-01:04:44]
B演示：一个备货需求发了部分，下次再发另一部分，「在经典里面找到这个订单下推已发货的」

### 拆单场景

原文 [05:06-06:00] + [22:37-22:42]：
- 一个订单有多台设备发给不同客户（常见于平台订单）
- B确认：经常会有拆单需求
- T的解决方案：通过附件（发货清单）来拆分，一个中台订单对应多个发货通知

## 三、库存的处理流程和判断

### 库存判断的完整决策树

原文 [13:17-15:23] 梳理出的流程：

```
订单来了
  ├→ 海外单 → 查海外仓库存
  │   ├→ 够 → 直接发
  │   └→ 不够 → 找销售借货（找亚马逊借）
  │            ├→ 借到了 → 发
  │            └→ 借不到 → 看能否从国内发
  │                       ├→ 能 → 确认涛哥运费够不够
  │                       │     ├→ 够 → 发
  │                       │     └→ 不够 → 销售调PI价格
  │                       └→ 不能 → 生产备货
  └→ 国内单 → 查国内库存
      ├→ 够 → 发
      └→ 不够 → 生产备货
               └→ 备好后再发
```

### 库存数据源现状

原文 [43:17-43:36]：
- B：「他给我的是海外仓的库存。国内的我没有」
- T：「我知道他也没有，国内的我跟金总商量」

**结论：一期只有海外仓库存可用，国内仓库存待确认数据源。**

### 分仓发货决策

原文 [14:03-15:23]：
- 海外单：B先确认从哪个海外仓发、库存够不够
- 国内单：B确认从武汉仓还是其他仓发
- 特殊需求（需报关等）→ 从国内发
- B在提单时会注明哪个单从哪个仓发

### 生产备货触发

原文 [12:37-12:42] + [15:37-15:47]：
- 订单推给仓库张燕 → 排哪些先发
- 没货时 → 生产备货
- B：「我下的单就相当于是给他的需求」
- T：「所以直接到生产那边」

## 四、更新机制（下推）

### "下推"的含义

金蝶概念：发货后标记订单已出库/已发货的状态更新操作。

原文 [25:28-25:35]：
T：「我们就把它下推了。市场那边告诉你已经发货了，他就发邮件，然后你进来打勾，推下去」
B：「每一次发货都会发邮件告诉我们发了，然后我们就搜到那个订单，做一个下推的动作」

### 当前的下推流程

```
物流发货 → 发邮件通知所有人
       → B收到邮件 → 搜订单 → 手工下推
```

### 一期 vs 二期边界

原文 [48:51-49:57]：
B问：「下推这一步都你做吗？」
T答：「我可以……但你下推这个过程，下周必须要你们自己搞。下推是另外一个流程，我觉得下一次再做可以」
B：「可以」

**一期：制单(Save) + 提交(Submit)**
**二期：下推（发货后自动标记）**

### 多次下推（分批发货）

原文 [23:40-24:06] + [25:28-25:35]：
- 一个订单可能分多次发货 → 每次发完都做一次下推
- B展示：一个2月的单，2月28号已全出；但有的单可能2月推了几台，过一阵又发一次又推一次
- 每个发货通知单对应一次下推

### 补单的特殊下推

原文 [01:01:18-01:02:21] B 讲解了调货场景的补充：
- 场景：独立站缺货需求下了10台，但不是马上推
- 而是：把这10台推到5月28号的补单里
- 同时：5月28号的补单里要把这2台去掉（避免重复）
- 然后：在22号的日期上做下推

**结论：补单下推可以跨日期、跨单号关联，不是简单的"发货了就下推"。**

## 总结：四个细节的相互关联

```
                    ┌─────────────────────┐
                    │   多公司主体调货      │
                    │ 深圳账号 / 香港账号   │←──── 决定了ERP制单用哪个FSaleOrgId
                    └──────┬──────────────┘
                           │ 库存不够时调货
                           ▼
                    ┌─────────────────────┐
                    │   库存判断决策树      │
                    │ 海外仓→够直接发       │←──── 决定了是否触发生产备货
                    │ 海外仓→不够→借货/调仓 │
                    └──────┬──────────────┘
                           │ 一个单多个客户/多批次
                           ▼
                    ┌─────────────────────┐
                    │   拆单/改单判断       │
                    │ 已闭环→新建           │←──── 决定了是改原单还是建新单
                    │ 未闭环→原单叠加       │
                    │ 已部分发货→区分处理    │
                    └──────┬──────────────┘
                           │ 发货后
                           ▼
                    ┌─────────────────────┐
                    │   下推(更新机制)      │
                    │ 一期: 制单+提交       │←──── 决定了系统边界
                    │ 二期: 自动下推        │
                    └─────────────────────┘
```
